# Semantic Search

**Status:** ✅ Implemented

**ID:** `search.semantic`

## Embedding models

### Model presets

> `{#search.semantic::presets}`

Four presets are available:

| Preset                         | Dimensions | Has prefixes |
| ------------------------------ | ---------- | ------------ |
| `nomic-embed-text-v1.5`       | 768        | yes          |
| `qwen3-0.6b`                  | 1024       | no           |
| `qwen3-4b`                    | 2560       | no           |
| `jina-embeddings-v2-base-code` | 768        | yes          |

Each preset specifies a Hugging Face repo and a GGUF filename. All models use
GGUF format for use with node-llama-cpp.

### Default preset

> `{#search.semantic::default-preset}`

When no `embedding.preset` is configured, the system uses
`nomic-embed-text-v1.5`.

### Preset dimensions

> `{#search.semantic::preset-dimensions}`

Each preset defines a fixed vector dimension. The search index schema uses this
dimension to create the vector field (`vector[N]`). Embeddings generated by a
preset always have exactly that number of dimensions.

### Preset prefixes

> `{#search.semantic::preset-prefixes}`

When a preset defines prefixes, it always defines both `queryPrefix` and
`documentPrefix`. A preset never defines only one of the two.

### Unknown preset rejection

> `{#search.semantic::unknown-preset}`

Given a preset name that does not match any known preset and no explicit `repo`
is configured, the system throws an error listing the available preset names.

### Model download on first run

> `{#search.semantic::model-download}`

On first use, the model file is resolved from
`hf:{repo}/{filename}`. If the file is not present locally, it is downloaded
to the models directory. A progress callback is invoked with `(downloaded, total)`
during download.

### Lazy initialization

> `{#search.semantic::lazy-init}`

The embedding model is not loaded until the first call to `embedQuery` or
`embedDocument`. Calling `initialize()` multiple times is idempotent — the model
is loaded only once.

## Embedding cache

### Cache keyed by content hash

> `{#search.semantic::cache.content-hash}`

Embeddings are cached by SHA-256 hash of the content that was actually embedded.
The hash is a 64-character lowercase hex string. Identical content always
produces the same hash; different content produces different hashes.

### Cache isolation per model

> `{#search.semantic::cache.model-isolation}`

Each embedding model has its own separate cache database
(`.ts-graph-mcp/embedding-cache/{model-name}.db`). An embedding cached under
model A is not visible when querying with model B.

### Cache persistence across restarts

> `{#search.semantic::cache.persistence}`

Embedding cache data persists across server restarts. Vectors stored in a
previous session are available in subsequent sessions without re-embedding.

### Cache hit returns stored vector

> `{#search.semantic::cache.hit}`

When the content hash of the text to embed matches an entry in the cache, the
stored vector is returned without calling the embedding model.

### Cache miss triggers embedding generation

> `{#search.semantic::cache.miss}`

When the content hash is not found in the cache, the embedding model generates
the vector. The result is stored in the cache for future lookups.

### Batch cache lookup

> `{#search.semantic::cache.batch}`

Multiple embeddings can be looked up in a single batch call. The result contains
only hashes that were found; missing hashes are silently omitted.

## Query embedding vs document embedding

### Query prefix

> `{#search.semantic::prefix.query}`

When a model preset defines a `queryPrefix`, it is prepended to the input text
before generating a query embedding. For example, with `nomic-embed-text-v1.5`,
the text `"validation"` is embedded as `"search_query: validation"`.

### Document prefix

> `{#search.semantic::prefix.document}`

When a model preset defines a `documentPrefix`, it is prepended to the input
text before generating a document embedding. For example, with
`nomic-embed-text-v1.5`, the text `"function validate() {}"` is embedded as
`"search_document: function validate() {}"`.

### No prefix when unset

> `{#search.semantic::prefix.none}`

When a preset does not define prefixes (e.g., `qwen3-0.6b`), the text is
embedded as-is, without any prefix, for both query and document embeddings.

## Vector similarity

### Cosine similarity metric

> `{#search.semantic::cosine}`

Vector similarity is measured using cosine similarity. Identical vectors produce
a score of 1. Orthogonal vectors produce a score of 0. Zero vectors produce a
score of 0.

### Cosine magnitude invariance

> `{#search.semantic::cosine.magnitude-invariance}`

Cosine similarity is independent of vector magnitude. Scaling both vectors by any
positive factor does not change the result.

### Vector similarity floor

> `{#search.semantic::similarity-floor}`

Vector search results are filtered by a minimum cosine similarity of 0.6.
Documents with a cosine similarity below this threshold are excluded from the
vector search results.

## Embedding fallback strategy

### Fallback: full content

> `{#search.semantic::fallback.full}`

The first embedding attempt uses the full prepared content (metadata prefix +
source snippet). If it fits within the model's context window, this embedding is
used.

### Fallback: stripped class implementation

> `{#search.semantic::fallback.stripped-class}`

If the full content overflows the model's context and the node type is `Class`,
the system retries with method bodies replaced by `{ ... }`. This strategy is
not applied to non-Class node types.

### Fallback: progressive truncation

> `{#search.semantic::fallback.truncation}`

If previous strategies overflow, the snippet is repeatedly halved until it either
fits or falls below a minimum useful length (100 characters).

### Fallback: metadata only

> `{#search.semantic::fallback.metadata-only}`

As a last resort, the system embeds only the metadata line
(`// {Type}: {name}\n// File: {path}`), without any source snippet. If even this
fails, the system throws an error.

### Non-overflow errors propagate

> `{#search.semantic::fallback.propagate-errors}`

Only context overflow errors trigger the fallback chain. Other errors (network
failures, etc.) are thrown immediately without attempting further strategies.

## Content preparation

### Content preparation

> `{#search.semantic::content-preparation}`

Before embedding, content is prepared by combining metadata and source snippet:
```
// {NodeType}: {name}
// File: {filePath}

{sourceSnippet}
```
The result is trimmed. This format is used for all document embeddings.
